# Document writing style

산문 문서 — README, 가이드, Markdown 문서, Jira 이슈, Confluence 페이지, Linear 코멘트 — 작성·수정 시 따르는 규칙. 코드 주석·커밋 메시지·PR 제목/설명은 대상이 아니다 ([language.md](./language.md) 참고).

## 언어

- 기본 한국어. 본문·헤더·불릿 모두 한국어.
- 영어 원문 유지:
  - 식별자: API·함수·클래스·변수명, 명령어, 파일 경로
  - 표준 기술 용어: cache hit, throughput, kernel, schematic, MCTP, PCIe 등
  - 고유명사: 제품명·회사명·프로토콜명
- 음차 표기("쓰루풋", "캐시 히트율") 피하고 영어 원어 사용.
- 영어 용어 뒤에 한국어 번역 괄호 병기 금지. 독자가 안다고 가정.
- 예외: `agents/` 레포 자체 문서(`AGENTS.md`, `instructions/*.md`, 레포 README 류)는 tool-neutral·다중 에이전트 컨텍스트라 영어로 둔다.

## 톤

- 간결·주장적. 결론과 행동을 먼저, 배경은 뒤에.
- 수식어·자기칭찬 제거. "매우", "정말", "강력한", "혁신적인", "포괄적인", "powerful", "comprehensive" 류 금지.
- 마케팅 투 금지. 사실과 근거로만.
- 능동태·명령조 선호. "X를 하면 좋습니다" → "X를 한다" 또는 "X를 해라".
- 완충 표현("~인 것 같습니다", "아마도", "조금") 제거.

## 서술 방식

- 가능하면 개조식으로 쓴다. 긴 설명이 꼭 필요할 때만 산문 단락.
- 불릿·넘버링으로 계층 구조를 표현. 같은 들여쓰기 레벨 = 같은 의미적 무게.
- 넘버링은 순서·우선순위가 있을 때만. 단순 나열은 불릿.
- 한 불릿이 두세 줄을 넘기면 더 잘게 쪼개거나 하위 불릿으로 분해할 수 있는지 검토.
- 줄바꿈 규칙: 한 단락·한 불릿 = 물리적 한 줄. 80/100자 wrap 같은 코드 컨벤션을 산문에 적용하지 않는다. 화면에서는 어차피 자동 wrap된다. 줄바꿈은 항목 구분·코드 블록·인용문 분리처럼 의미가 있을 때만.

## 강조 절제

- **bold**, `code`, *italic* 모두 남발 금지. 한 화면에 강조가 여러 개면 이미 과한 것.
- `code` 백틱은 다음에만 사용:
  - 식별자(함수·변수·클래스·파일 경로·명령어·플래그)
  - 정확한 문자열 리터럴
- **bold**는 다음에만 사용:
  - 같은 단락 안에서 놓치면 안 되는 단어 한두 개
  - 표·정의 목록에서 용어 항목
- 헤더에는 bold를 쓰지 않는다 (헤더 자체가 이미 강조).
- 강조가 필요한지 한 번 더 자문한다. 빼도 의미가 통하면 뺀다.

## 구조 / 헤더

- 헤더 레벨은 계층적·일관적. 같은 레벨에 같은 의미적 무게의 항목만.
  - 같은 H2끼리 동일한 추상화 수준이어야 한다.
    - OK: "설치 / 사용법 / 테스트"
    - NOT OK: "설치 / 테스트 실행 시 환경변수" (후자는 H3 수준)
  - H1 → H2 → H3 계층을 따라 내려간다. 중간 레벨 건너뛰기 금지.
  - 같은 부모 아래 자식 헤더가 하나뿐이면 헤더 빼고 본문에 흡수.
- 짧은 문서는 헤더 계층을 과하게 만들지 않는다. 한두 단락이면 헤더 없이.
- 헤더는 명사구 또는 짧은 동사구. 서술형 헤더("이번에 우리가 무엇을 했는가") 금지.

## 표

- 모든 셀에 padding을 넣어 파이프(`|`)가 세로로 정렬되게 작성한다. 에디터(렌더링 전 raw 뷰)에서도 읽힐 수 있어야 한다.
- 헤더 구분선(`---`)의 길이도 각 열의 최대 폭에 맞춘다.
- 좁은 셀은 공백으로 패딩, 넓은 셀은 그대로. 열 폭은 그 열에서 가장 긴 셀 기준.
- 예시:

  ```markdown
  | Command   | Purpose                          | Notes        |
  | --------- | -------------------------------- | ------------ |
  | `build`   | Compile firmware and drivers     | Long-running |
  | `package` | Bundle PetaLinux boot image      | Needs XSA    |
  | `commit`  | Format and stage HEAD commit msg | —            |
  ```

- 표 내용이 너무 길어 한 행이 화면을 넘기면 표 대신 불릿+하위 불릿으로 풀어쓰는 걸 고려한다.

## 다이어그램

- 단순한 그림(블록 박스, 직선 흐름, 트리)은 코드 블록 ASCII 아트로 그린다.

  ```
  ┌────────┐    ┌────────┐
  │ Client │ -> │ Server │
  └────────┘    └────────┘
  ```

- 복잡한 화살표·분기·상태 전이가 필요하면 Mermaid를 쓴다.

  ```mermaid
  flowchart LR
    A[Request] --> B{Auth?}
    B -- yes --> C[Handler]
    B -- no  --> D[401]
  ```

- 판단 기준: 화살표가 교차하거나, 분기/병합이 두 군데 이상이거나, 노드가 6개 넘으면 Mermaid. 그 외엔 ASCII.
- 이미지 파일 첨부는 가능하면 피한다. 텍스트로 그릴 수 있으면 텍스트로.

## 금지 사항

- 이모지·이모티콘 일체 금지. ✅ ⚡ 🚀 ✨ 📝 등 장식용, 콜아웃 아이콘 모두 포함. 사용자가 명시 요청하지 않는 한 절대 넣지 않는다.
- 끝맺음 요약 단락 금지. "Summary", "마무리", "In conclusion", "정리하자면", "이번에 우리는 ~를 했습니다" 류 재탕 요약을 마지막에 붙이지 않는다.
- 과장된 표현·마케팅 투 금지. "powerful", "comprehensive", "혁신적인", "강력한", "획기적인", "완벽한", "최첨단" — 사실로 뒷받침 안 되는 형용사는 뺀다.
- 자기 작업 자평("깔끔하게 정리했습니다", "잘 동작합니다") 금지. 결과만 적는다.
- 의무적 TL;DR 금지. 정말 긴 문서에서만 사용.

## 도구별 조정

- README / Markdown: 위 규칙 그대로. 코드 블록에 언어 태그 부착.
- Jira 이슈:
  - 제목은 한 줄 사실 진술.
  - 본문 최소 구조: 배경 / 목표 / 작업 항목 / 완료 조건(AC).
  - 헤더 남발 금지. 불릿으로 충분.
- Confluence:
  - 페이지가 길어지므로 H2 목차를 먼저 정한 뒤 채운다.
  - 같은 H2 레벨 항목들의 무게가 균일한지 작성 후 점검.
- Linear: 코멘트·설명은 짧은 한국어. 코드/링크 참조는 인라인.
